# 예외상황 테스트

실행 기준:
- 시나리오 하네스 기반 실험 모듈: `python3 -m tichu.test_scenario_edge_cases`
- 실행 결과: `scenario edge case experiments OK`

목적:
- 폭탄 끼어들기
- 폭탄 vs 폭탄 비교
- 참새 콜 강제 패스 차단
- 봉황 포함 예외 케이스

---

## 1. 다른 사람 턴에 폭탄을 낼 수 있는지

테스트 시나리오:
- 현재 턴: 플레이어 1
- 테이블: `S-8`
- 플레이어 3 손패:
  - 사폭탄: `S-9 H-9 D-9 C-9`
  - 스트레이트 플러시 폭탄: `S-5 S-6 S-7 S-8 S-9`

실험 결과:
- `preview_play` 기준 두 폭탄 모두 `combo_type`은 정상적으로 폭탄으로 해석됨
- 하지만 결과는 둘 다 `reason_code=NOT_CURRENT_PLAYER`
- 실제 `play` 시도 결과도 둘 다 `error.message = "not this player's turn"`

정리:
- 현재 구현은 폭탄을 "다른 사람 턴에 끼어들어 내는" 규칙을 지원하지 않음
- 즉, 폭탄 자체 판정은 되지만 턴 검사에서 먼저 막힘

---

## 2. 맨 바닥이나 개 사용 중 폭탄을 낼 수 있는지

### 2-1. 맨 바닥에서 다른 플레이어가 끼어들기

테스트 시나리오:
- 테이블 비어 있음
- 현재 턴: 플레이어 1
- 플레이어 3이 사폭탄 보유

실험 결과:
- `preview_play.reason_code = NOT_CURRENT_PLAYER`
- 즉, 맨 바닥에서도 비턴 플레이어의 폭탄 끼어들기는 불가

### 2-2. 개 사용 직후(리드가 팀원에게 넘어간 상태) 끼어들기

테스트 시나리오:
- 개 사용 후 리드가 플레이어 2에게 넘어간 상태
- 현재 턴: 플레이어 2
- 플레이어 3이 사폭탄 보유

실험 결과:
- `preview_play.reason_code = NOT_CURRENT_PLAYER`
- 즉, 개 사용으로 현재 트릭이 비어 있는 상태에서도 비턴 플레이어의 폭탄 끼어들기는 불가

### 2-3. 현재 턴 플레이어가 맨 바닥에 폭탄을 리드하는 경우

테스트 시나리오:
- 현재 턴: 플레이어 1
- 테이블 비어 있음
- 플레이어 1 손패: `S-9 H-9 D-9 C-9`

실험 결과:
- `preview_play.reason_code = OK`
- 실제 `play`도 성공

정리:
- 현재 구현은 "비턴 폭탄 끼어들기"는 막고 있음
- 하지만 "자기 턴에 맨 바닥으로 폭탄 리드"는 허용하고 있음

## 3. 폭탄을 더 높은 폭탄으로 덮을 수 있는지

테스트 범위:
- 사폭탄 vs 사폭탄
- 사폭탄 vs 스트레이트 플러시 폭탄
- 스트레이트 플러시 폭탄 vs 사폭탄
- 스트레이트 플러시 폭탄 vs 스트레이트 플러시 폭탄

실험 결과:

| 케이스 | 테이블 폭탄 | 선택 폭탄 | 결과 |
|---|---|---|---|
| `bomb_four_over_weaker_bomb_four` | `8x4` | `9x4` | 가능 |
| `bomb_four_over_stronger_bomb_four` | `10x4` | `9x4` | 불가 |
| `straight_flush_over_bomb_four` | `Qx4` | `S-5 S-6 S-7 S-8 S-9` | 가능 |
| `bomb_four_over_straight_flush` | `H-4 H-5 H-6 H-7 H-8` | `9x4` | 불가 |
| `straight_flush_over_weaker_straight_flush` | `H-4 H-5 H-6 H-7 H-8` | `S-5 S-6 S-7 S-8 S-9` | 가능 |
| `straight_flush_over_stronger_straight_flush` | `H-6 H-7 H-8 H-9 H-10` | `S-5 S-6 S-7 S-8 S-9` | 불가 |

정리:
- 사폭탄끼리는 숫자가 더 높은 쪽만 덮을 수 있음
- 스트레이트 플러시 폭탄은 사폭탄을 덮을 수 있음
- 사폭탄은 스트레이트 플러시 폭탄을 덮을 수 없음
- 스트레이트 플러시 폭탄끼리는 더 높은 쪽만 덮을 수 있음

즉, 현재 구현의 폭탄 비교 규칙은 아래처럼 동작함:
- `bomb_four < bomb_straight_flush`
- 같은 종류 폭탄끼리는 강도 비교

## 4. 콜이 걸려있을 때, 해당 숫자 조합이 있으면 pass가 막혀있는지

기준:
- 콜 숫자: `9`
- 모두 현재 턴 플레이어 기준 실험
- `available_actions.can_pass` 와 `legal_plays`를 같이 확인

실험한 경우의 수:

| 케이스 | 손패 예시 | 현재 테이블 | legal_plays 결과 | pass |
|---|---|---|---|---|
| `single_exact` | `S-9` | `S-8` | `S-9` | 막힘 |
| `pair_exact` | `S-9 H-9` | `D-8 C-8` | `S-9 H-9` | 막힘 |
| `pair_phoenix` | `S-9 PHX` | `D-8 C-8` | `S-9 PHX` | 막힘 |
| `triple_exact` | `S-9 H-9 D-9` | `8-8-8` | `9-9-9` | 막힘 |
| `triple_phoenix` | `S-9 H-9 PHX` | `8-8-8` | `9-9-PHX` | 막힘 |
| `full_house_exact` | `5-5 9-9-9` | `4-4 8-8-8` | `5-5 9-9-9` | 막힘 |
| `full_house_phoenix` | `5-5 9-9 PHX` | `4-4 8-8-8` | `5-5 9-9-PHX` | 막힘 |
| `straight_exact` | `7-8-9-10-J` | `3-4-5-6-7` | `7-8-9-10-J` | 막힘 |
| `straight_phoenix` | `7-8-10-J PHX` | `3-4-5-6-7` | `7-8-9-10-J(PHX)` | 막힘 |
| `pair_run_exact` | `8-8 9-9` | `6-6 7-7` | `8-8 9-9` | 막힘 |
| `pair_run_phoenix` | `8-8 9 PHX` | `6-6 7-7` | `8-8 9-9(PHX)` | 막힘 |

정리:
- 현재 구현에서는 "콜 숫자를 만족하는 합법 조합이 하나라도 있으면" `pass`가 막힘
- 위 실험 범위에서는 exact 조합과 봉황 대체 조합 모두 `can_pass = False`
- 즉, 현재 지원하는 콜 강제 로직은 아래 족보 범위에서 정상 동작함
  - single
  - pair
  - triple
  - full house
  - straight
  - pair run

## 5. 추가로 확인한 특수 케이스

### 5-1. 봉황이 있어도 실제로 콜 숫자를 만들지 못하면 pass 가능

테스트 시나리오:
- 손패: `S-8 PHX`
- 콜 숫자: `7`
- 테이블: `S-10`

실험 결과:
- `legal_plays = []`
- `available_actions.can_pass = True`

정리:
- 손패에 봉황이 있다는 사실만으로 콜 강제가 걸리지는 않음
- 봉황이 실제로 콜 숫자를 만족하는 합법 조합을 만들 수 있어야만 pass가 막힘

### 5-2. 현재 구현에서 폭탄은 "특수 끼어들기 액션"이 아니라 일반 play 규칙 안에서만 처리됨

관찰 결과:
- 폭탄 비교 자체는 `can_beat()`에서 잘 처리됨
- 하지만 실제 액션은 `play_cards()`의 현재 턴 검사(`not this player's turn`)를 먼저 통과해야 함

정리:
- 현재 구현 상태를 한 줄로 요약하면:
  - "폭탄의 족보/강도 비교는 구현됨"
  - "비턴 폭탄 끼어들기 규칙은 아직 미구현"

---

## 결론

- 폭탄 비교 규칙 자체는 잘 들어가 있음
- 하지만 "다른 사람 턴에 폭탄 끼어들기"는 아직 지원하지 않음
- 현재 구현은 자기 턴의 일반 `play` 흐름 안에서만 폭탄을 처리함
- 참새 콜 강제는 봉황 포함 주요 조합 케이스에서 `pass` 차단이 정상 동작함
