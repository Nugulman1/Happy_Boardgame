## 기술 스택 (확정 / 보류)

### 확정

- **Engine/Server 로직: Python 단일화**
    - 보드게임 구현이 “초정밀/고성능”까지는 당장 필요 없으므로 Python로 충분
    - 빠른 실험/디버깅/플러그인 로딩에 유리
- **멀티플레이 통신: WebSocket**
    - 처음부터 실시간 멀티플레이 가능한 구조로 가기 위해 WebSocket으로 고정
    - 서버 authoritative(서버가 룰/상태의 단일 진실)
- **PC 클라이언트 엔진: Godot**
    - 초기에는 테스트용으로만 최소 UI 구성
    - 보드게임 생성/실행 기능이 안정화되면 UI/엔진 확장

### 보류(향후)

- **DB: PostgreSQL**
    - MVP 단계에서는 메모리 세션/파일 로그로 시작 가능
    - 운영/리플레이 저장/매치 히스토리 필요해지면 도입

## 기능 개발 우선순위

### 1) 보드게임 구현 기능부터

- “게임 1개 완성”보다 먼저, **어떤 게임이든 끼워 넣을 수 있는 공통 템플릿 + 게임별 플러그인 구조**를 만든다.
- 게임별 특색은 플러그인으로 분리하고, 공통 엔진은 고정한다.

### 2) 1차 타겟 게임

- 초기 샘플 게임으로 **티츄를 선정함.**

---

## 구현 파이프라인 (핵심)

### A. 룰 → 공통 요소 정리 → 검수 → 플러그인 코드 생성

- **보드게임 룰(자연어)**
    
    → **1단계 프롬프트**: LLM이 보드게임 공통 요소(플레이어, 턴, 행동, 자원, 조건 등) 정리
    
    → **사용자 검수**: 정리 결과 확인·수정
    
    → **2단계 프롬프트**: 검수된 내용을 LLM에 넘겨 게임 플러그인(서버 로직) 코드 생성
    
    → 실행 및 테스트
    

- **장점**: IR(JSON) 중간 변환 없이 자연어·정리본만 사용 → LLM 비용·복잡도 감소
- **프롬프트**: `docs/prompts/01-룰-공통요소정리.md`, `02-정리내용-코드생성.md`

## 공통 템플릿(엔진) vs 게임별 구현(플러그인) 분리

### 공통 엔진이 담당(고정)

- 게임 상태(State) 저장/관리
- 플레이어/턴 관리(기본 구조)
- 액션 수신(WebSocket) 및 브로드캐스트
- 이벤트 로그(리플레이/디버깅)
- 테스트 러너(샘플 시나리오 실행)

### 게임 플러그인이 담당(게임별)

- 게임 규칙(검증/적용)
- 상태 초기화
- 종료/승리 조건
- 게임 고유 액션 정의

> 원칙: LLM/사람이 만든 코드는 “플러그인 영역”만 건드린다. 공통 엔진은 최대한 변하지 않게 유지한다.
>

## LLM 사용 방식(중요 결정)

- 런타임에서 서버가 LLM을 직접 호출하지 않는다.
- **보드게임을 만들기 전에** LLM에게 "룰→공통 요소 정리" / "정리본→플러그인 코드"를 묻고,
    
    그 결과물(정리본/플러그인 코드)을 사용자가 검수한 뒤 시스템에 **주입**해서 실행/검증한다.
    
- 시스템의 핵심은 “생성”이 아니라:
    - 사용자 검수(1단계 정리본)
    - 플러그인 로딩
    - 실행/테스트/로그
    - 멀티플레이 동기화(WebSocket)
