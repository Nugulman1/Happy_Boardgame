- **현재 핵심 축**은 `app/` HTTP API, `tichu/` 규칙 로직, `godot/` 테스트용 클라이언트다.
- **세션 저장 방식**은 아직 메모리 기반이다. `app/tichu_api.py`의 `_sessions` 딕셔너리에 `GameSession`을 보관하며, `docker-compose.yml`의 PostgreSQL은 준비만 되어 있고 현재 게임 세션 저장에는 쓰지 않는다.
- **문서 위치 주의**: 구현 메모는 `docs/`가 아니라 `tichu/구현_메모.md`에 있다.
- **프롬프트 폴더**는 현재 `01-룰-공통요소정리.md`만 있다. 요약/생성 2단계 문서는 아직 저장소에 없다.

## HTTP API 계층 (`app/`)

- **진입점**: `app/main.py`
  - `/`, `/health` 기본 라우트와 `register_tichu_api(app)` 호출만 둔다.
- **API 구현**: `app/tichu_api.py`
  - `GameSession`은 `game_id`, `state`, `round_state`, `phase`, `grand_tichu_responses`, `exchange_choices`, `last_round_result`를 묶는다.
  - 응답은 `game_id`, `viewer`, `phase`, `state`, `available_actions`를 기본으로 하고, 상황에 따라 `effects`, `round_result`를 추가한다.
  - 준비 단계(`prepare_grand_tichu`, `prepare_exchange`)에서는 `leader_index`, `current_player_index`를 `null`로 내려준다.
  - `viewer`는 조회 시점, `player_index`는 실제 행동 주체로 분리했다.
- **엔드포인트**
  - 생성/조회: `POST /games/tichu`, `GET /games/{game_id}?viewer=...`
  - 준비 단계: `POST /games/{game_id}/prepare/grand-tichu`, `POST /games/{game_id}/prepare/exchange`
  - 액션: `POST /games/{game_id}/actions/small-tichu`, `.../play`, `.../pass`
  - 용 수령자: `POST /games/{game_id}/actions/dragon-recipient`
  - 보조 API: `GET /games/{game_id}/legal-plays`, `POST /games/{game_id}/preview-combo`
- **에러 형식**
  - `ApiError`와 validation handler로 `400/404/409` + `{error: {code, message}}` 형식으로 맞춘다.
- **테스트**: `app/test_tichu_api.py`
  - 게임 생성, viewer 직렬화, 준비 단계 집계, 플레이/패스, 용 수령자, helper API, 자동 다음 라운드, 게임 종료를 검증한다.

## 티츄 인게임 로직 (`tichu/`)

- **상태 분리**: `tichu/state.py`
  - `GameState`는 외부 게임 상태(`team_scores`, `round_index`)만 유지한다.
  - `RoundState`는 손패, 트릭 상태, 티츄 선언, 참새 콜, 획득 더미, 탈락 순서 등 라운드 전용 데이터를 가진다.
- **준비 단계**: `tichu/prepare.py`
  - 초기 8장 배분, 나머지 6장 배분, 교환 적용, 참새 선 결정, 첫 트릭 시작을 담당한다.
- **족보/비교**: `tichu/combo_info.py`
  - 일반 족보, 특수 카드, 봉황 포함 조합, 폭탄 판정과 비교 규칙을 담당한다.
- **참새 콜**: `tichu/mahjong_call.py`
  - 참새 콜 저장/해제, 콜 강제 여부, 패스 가능 여부, 합법 조합 탐색을 분리해 둔다.
- **트릭/라운드 진행**: `tichu/trick.py`
  - `play_cards`, `pass_turn`, `resolve_trick_end`, `declare_small_tichu`, `is_round_over` 등 실제 진행 규칙을 처리한다.
  - 개, 용 수령자 선택, 봉황 싱글 비교 강도 유지까지 포함한다.
- **점수 처리**: `tichu/scoring.py`
  - 원투 승리와 일반 종료를 나눠 팀 점수 변화량을 계산하고 누적한다.
- **게임/세션 오케스트레이션**
  - `tichu/game_loop.py`: `run_game()` 뼈대와 `finalize_round()` 제공
  - `tichu/session_service.py`: 현재 HTTP API가 실제로 사용하는 서비스 계층
  - `tichu/engine.py`: 내부 엔진 v1. `start_round`, `submit_action`, `EngineSnapshot` 기반으로 동작하며 현재 FastAPI/Godot 경로와는 분리된 테스트용 런타임이다.
- **테스트**
  - `test_phase3.py`: 족보/폭탄/비교 규칙
  - `test_phase4.py`: 참새 콜
  - `test_phase5_trick_core.py`: 트릭 상태전이
  - `test_phase6_round_scoring.py`: 라운드 종료/점수
  - `test_engine_v1.py`: 내부 엔진 v1 phase 흐름

## Godot 1차 클라이언트 (`godot/`)

- **구조**
  - `scenes/Main.tscn`: 단일 테스트 화면
  - `scripts/ApiClient.gd`: HTTP 요청 래퍼
  - `scripts/GameStore.gd`: 최신 snapshot, `selected_viewer`, `selected_actor` 저장
  - `scripts/Main.gd`: UI 이벤트 처리와 화면 갱신
- **현재 연동된 기능**
  - `/health`
  - 새 게임 생성 / 상태 새로고침
  - viewer / actor 분리 테스트
  - grand tichu
  - exchange
  - small tichu
  - play / pass
  - dragon recipient 선택
  - round_result / 자동 다음 라운드 / game over 표시
- **아직 덜 붙은 부분**
  - `play`의 `call_rank` 입력과 `mahjong_call_rank` 표시
  - `state.players`, `leader_index`, `trick_index`, `players_out_order` 기반 요약 UI
  - `legal-plays`, `preview-combo` 보조 API
  - `effects` 로그/간단 이벤트 표시
  - 버튼 활성/비활성의 전면적인 `available_actions` 기준 정렬
