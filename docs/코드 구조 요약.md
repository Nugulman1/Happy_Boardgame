## API 계층 (`app/`)

### 진입점

- [app/main.py](/home/nugulman/dev/Happy_Boardgame/app/main.py)
  - `/`, `/health` 기본 라우트와 `register_tichu_api(app)` 호출만 둔다.

### API 구현

- [app/tichu_api.py](/home/nugulman/dev/Happy_Boardgame/app/tichu_api.py)
  - `_sessions`: `game_id -> GameSession`
  - `_rooms`: `room_code -> RoomSession`
  - `_game_to_room`: room-backed 게임의 역참조
  - `SocketManager`: 게임 snapshot/action용 WebSocket 연결 관리
  - `RoomSocketManager`: 로비 snapshot용 WebSocket 연결 관리
  - `SeatState`, `RoomSession`: 멀티플레이 로비/좌석 상태 모델

### 게임 API

- 생성/조회
  - `POST /games/tichu`
  - `GET /games/{game_id}?viewer=...&seat_token=...`
- 실시간 인게임 연결
  - `WS /ws/games/{game_id}?viewer=...&seat_token=...`
- 준비 단계
  - `POST /games/{game_id}/prepare/grand-tichu`
  - `POST /games/{game_id}/prepare/exchange`
- 액션
  - `POST /games/{game_id}/actions/small-tichu`
  - `POST /games/{game_id}/actions/play`
  - `POST /games/{game_id}/actions/pass`
  - `POST /games/{game_id}/actions/dragon-recipient`
- helper API
  - `GET /games/{game_id}/legal-plays`
  - `POST /games/{game_id}/play-preview`

### WebSocket 액션 프로토콜

- client -> server
  - `{"type":"action","request_id":1,"action":"play","payload":{...}}`
- server -> requester
  - 성공: `action_result`
  - 실패: `action_error`
- server -> 연결된 viewer 전체
  - `snapshot`
- 상태 반영의 단일 소스는 여전히 `snapshot`이다.

### room / seat token 멀티플레이

- room API
  - `POST /rooms/tichu`
  - `POST /rooms/{room_code}/join`
  - `GET /rooms/{room_code}?seat_token=...`
  - `POST /rooms/{room_code}/start?seat_token=...`
  - `POST /rooms/{room_code}/leave?seat_token=...`
  - `WS /ws/rooms/{room_code}?seat_token=...`
- room 생성자는 자동으로 `seat 0` host가 된다.
- room-backed 게임은 `seat_token -> player_index` 인증을 강제한다.
- direct dev/test 게임(`POST /games/tichu`)은 기존처럼 seat token 없이 동작한다.

### 현재 설계 포인트

- 액션 입력은 WebSocket, helper API는 HTTP 유지다.
- room socket은 로비 전용, game socket은 인게임 전용으로 분리했다.
- `viewer`는 room-backed 게임에서는 사실상 인증된 좌석 번호다.
- 재접속은 같은 `seat_token`과 같은 로컬 세션 파일을 전제로 한다.

- [godot/scripts/Main.gd](/home/nugulman/dev/Happy_Boardgame/godot/scripts/Main.gd)
  - 방 생성/입장/시작/나가기
  - `user://multiplayer_session.cfg` 저장/복구
  - room socket -> game socket 전환
  - 액션 버튼 WebSocket 송신
  - snapshot 기준 화면 갱신
  - presentation event consumer
  - 현재 연결된 임시 연출: 선언 배너, 카드 이동 문구, 현재 턴 강조, 라운드 점수 요약

### 현재 동작

- 방 생성 후 seat 0 host 자동 점유
- room code로 다른 클라이언트가 seat 1~3 입장
- host 시작 후 인게임 socket으로 전환
- 각 클라이언트는 자기 seat 기준 snapshot만 본다.
- 로컬 세션 파일이 남아 있으면 앱 시작 시 room/game 복구를 시도한다.
- snapshot의 `effects`는 `EffectInterpreter -> PresentationBus -> Main.gd consumer` 순서로 처리된다.
- `turn_changed`는 플레이어 요약 텍스트에 `<TURN>` 표시로 반영된다.
- `grand_tichu_declared`, `small_tichu_declared`는 상단 announcement 라벨에 임시 배너로 반영된다.
- `cards_played`, `trick_won`은 table motion 라벨로 임시 카드 이동/회수 연출을 대신한다.
- `round_finished`는 결과 라벨과 별도 점수 요약 라벨로 임시 표시된다.

### 아직 남은 부분

- 실제 카드 이동 애니메이션, 사운드, 캐릭터 리액션은 아직 없다.
- 현재 연출은 텍스트 placeholder 위주라 나중에 실제 모션/SFX consumer로 교체해야 한다.
- 3D 캐릭터 자산, 리깅, 애니메이션은 아직 프로젝트에 없다.
- room/game 로직이 `Main.gd`, `tichu_api.py` 한 파일에 몰려 있어 다음 단계에서 분리 여지가 있다.

## 테스트 상태

- [app/test_tichu_api.py](/home/nugulman/dev/Happy_Boardgame/app/test_tichu_api.py)
  - 규칙 API
  - WebSocket snapshot/action
  - room 생성/입장/시작/leave
  - seat token 인증
  - room/game 재접속
- 최근 기준 서버 테스트는 `20 passed`까지 확인했다.
- Godot는 Linux 실행 파일로 headless 로드가 되며, 최근 presentation bus/effect interpreter 추가 후에도 프로젝트 로드가 통과한다.
- 멀티플레이 흐름과 presentation event 소비는 여전히 수동 테스트 비중이 크다.
