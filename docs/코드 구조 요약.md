# 코드 구조 요약

## 프로젝트 구조

```
Boardgame/
  app/
    main.py           # FastAPI 진입점 (/, /health). Docker에서 uvicorn으로 실행.
  games/
    _template/
      game.yaml       # 게임 설정 템플릿. 새 게임 시 games/{game_id}/game.yaml 로 복사.
  docs/
    plan/             # 단계별 구현 계획 및 임시 설계 메모
      봉황_임시.md
      폭탄_임시.md
      참새_콜_임시.md
    prompts/          # LLM 프롬프트 템플릿 (파이프라인 1·2단계)
      01-룰-공통요소정리.md   # 룰(자연어) → 공통 요소 정리 (사용자 검수용)
      02-정리내용-코드생성.md # 검수된 정리본 → 플러그인 코드 생성
    TODO.md
    보드게임플랫폼기술_방향성.md
    코드 구조 요약.md  # 이 문서
```

- **Docker**: `docker-compose up --build` 로 app(Python API) + db(PostgreSQL) 실행. 엔진은 앱 밖/추가 예정.
- **파이프라인**: 룰(자연어) → 1단계 프롬프트(공통 요소 정리) → 사용자 검수 → 2단계 프롬프트(플러그인 코드 생성). IR(JSON) 중간 변환 없음. `docs/prompts/` 참고.

## 티츄 인게임 로직 (tichu/)

- **공통 설계**: `docs/공통게임로직_설계.md` — 진행 순서대로 로직 구성, 단계별 구현, 메인 while 하나.
- **상태**: `GameState`(외부: team_scores, round_index) / `RoundState`(내부: hands, deck, leader_index, trick_index, grand_tichu_declarers, current_trick_cards). 라운드마다 `new_round_state()`로 생성.
- **메인 루프**: `tichu/game_loop.py` — `run_game(state)` → `init_state` 후 `while not is_game_over(state)`: `round_state = new_round_state()`, `run_prepare_phase(round_state)`, (3단계~ placeholder).
- **준비 단계**: `tichu/prepare.py`
  - `sort_hand`, `deal_initial_8`, `deal_remaining_6` → `apply_exchange(round_state, choices)` → `set_leader_by_mahjong` → `run_prepare_phase(round_state)` 한 번에 실행.
  - 라지 티츄·교환은 **프론트에서 데이터 주면** `round_state`에 반영 후 다음으로 넘어가는 식. `run_prepare_phase`는 인자 없이 `round_state`만 받음.
  - 교환: 프론트에서 4명분 `choices` 받으면 `apply_exchange(round_state, choices)` 호출. choices는 `(to_left, to_team, to_right)` 4개.
- **족보/비교 로직**: `tichu/combo_info.py`
  - `ComboInfo`: 카드 묶음을 해석한 결과(`card_count`, `combo_type`, `strength`, `uses_phoenix`).
  - `evaluate_combo(cards)`: 카드 목록을 바로 해석해 싱글/페어/트리플/풀하우스/스트레이트/연속페어를 판정.
  - 현재는 참새, 용 싱글, 봉황 일반 족보를 지원하고, 폭탄/개/콜은 후속 단계 예정.
  - `can_beat(current_combo, selected_combo)`: 같은 종류 족보 비교를 담당. 봉황 싱글은 직전 싱글 기준 `+0.5` 규칙 반영.
- **테스트**: `tichu/test_phase1.py`, `tichu/test_phase3.py`
  - 1·2단계 기본 검증과 3단계 족보 판정/비교 검증을 분리해 유지.
- **구현 메모**: `tichu/구현_메모.md` — 카드 숫자, 점수, State 분리, 단계별 구현 메모.
