# 코드 구조 요약

## 프로젝트 구조

```
Boardgame/
  app/
    main.py           # FastAPI 진입점 (/, /health). Docker에서 uvicorn으로 실행.
  games/
    _template/
      game.yaml       # 게임 설정 템플릿. 새 게임 시 games/{game_id}/game.yaml 로 복사.
  docs/
    plan/             # 단계별 구현 계획 및 임시 설계 메모 폴더 (현재 비어 있음)
    prompts/          # LLM 프롬프트 템플릿 (파이프라인 1·2단계)
      01-룰-공통요소정리.md   # 룰(자연어) → 공통 요소 정리 (사용자 검수용)
      02-정리내용-코드생성.md # 검수된 정리본 → 플러그인 코드 생성
    TODO.md
    보드게임플랫폼기술_방향성.md
    코드 구조 요약.md  # 이 문서
```

- **Docker**: `docker-compose up --build` 로 app(Python API) + db(PostgreSQL) 실행. 엔진은 앱 밖/추가 예정.
- **파이프라인**: 룰(자연어) → 1단계 프롬프트(공통 요소 정리) → 사용자 검수 → 2단계 프롬프트(플러그인 코드 생성). IR(JSON) 중간 변환 없음. `docs/prompts/` 참고.

## 티츄 인게임 로직 (tichu/)

- **공통 설계**: `docs/공통게임로직_설계.md` — 진행 순서대로 로직 구성, 단계별 구현, 메인 while 하나.
- **상태**: `GameState`(외부: `team_scores`, `round_index`) / `RoundState`(내부: `hands`, `deck`, `leader_index`, `current_player_index`, `trick_index`, `grand_tichu_declarers`, `current_trick_cards`, `current_trick_pile`, `last_played_by`, `pass_count_since_last_play`, `mahjong_call_rank`, `small_tichu_declarers`, `played_first_card_players`, `won_trick_cards`, `players_out_order`). 라운드마다 `new_round_state()`로 생성.
- **메인 루프**: `tichu/game_loop.py`
  - `run_game(state)`는 아직 자동 플레이 루프가 아니라 `init_state` 후 새 라운드 상태를 만들고 준비 단계까지만 진행하는 뼈대다.
  - 실제 트릭 액션은 외부가 `play_cards`/`pass_turn`로 넣고, 라운드 종료 뒤 `finalize_round(state, round_state)`로 점수를 누적하는 구조다.
- **준비 단계**: `tichu/prepare.py`
  - `sort_hand`, `deal_initial_8`, `deal_remaining_6` → `apply_exchange(round_state, choices)` → `set_leader_by_mahjong` → `run_prepare_phase(round_state)` 한 번에 실행.
  - 라지 티츄·교환은 **프론트에서 데이터 주면** `round_state`에 반영 후 다음으로 넘어가는 식. `run_prepare_phase`는 인자 없이 `round_state`만 받음.
  - 교환: 프론트에서 4명분 `choices` 받으면 `apply_exchange(round_state, choices)` 호출. choices는 `(to_left, to_team, to_right)` 4개.
  - 준비 단계 마지막에 `start_trick(round_state)`를 호출해 첫 트릭 상태까지 맞춘다.
- **트릭/라운드 로직**: `tichu/trick.py`
  - `play_cards` / `pass_turn` / `resolve_trick_end`가 현재 차례, 트릭 상태, 손패 제거, 트릭 회수, 다음 선을 처리한다.
  - `can_declare_small_tichu`, `declare_small_tichu`로 첫 카드 제출 전 스몰 티츄 선언을 저장한다.
  - `is_double_victory`, `is_round_over`, `get_round_end_reason`로 원투와 3명 소진 기반 라운드 종료를 판정한다.
  - 개는 `combo_type="dog"`로 판정하고, 맨바닥 + 자기 첫 턴일 때만 낼 수 있다. 내면 트릭을 만들지 않고 같은 팀 플레이어에게 즉시 선을 넘긴다.
  - 용으로 트릭을 잡으면 `dragon_recipient`로 지정한 상대 팀 플레이어에게 해당 트릭 더미를 넘긴다.
- **참새 콜 로직**: `tichu/mahjong_call.py`
  - `can_declare_mahjong_call`, `set_mahjong_call`, `clear_mahjong_call`로 참새 포함 합법 조합을 낸 순간 콜(없음 포함)을 저장/해제한다.
  - `hand_can_possibly_match_mahjong_call`로 손패에 콜 숫자 또는 봉황이 있는지만 먼저 빠르게 검사한다.
  - `find_legal_plays`는 손패의 non-empty subset을 전부 돌며 `evaluate_combo`와 `can_beat`로 현재 트릭 기준 합법 조합을 찾는다. 현재는 참새 콜 강제 여부 판단 쪽에서 주로 쓴다.
  - `selection_satisfies_mahjong_call`은 실제 콜 숫자 카드 또는 `resolved_ranks`를 사용해 봉황 대체까지 포함한 콜 충족 여부를 판정한다.
  - `find_legal_plays_matching_call`, `must_follow_mahjong_call`, `can_pass_with_mahjong_call`으로 콜 강제와 패스 가능 여부를 순수 함수로 분리했다.
- **점수 로직**: `tichu/scoring.py`
  - `calculate_round_score_deltas(round_state)`가 라운드 종료 후 팀 점수 변화량을 계산한다.
  - 원투면 카드 점수는 무시하고 기본 200점 + 티츄 선언 점수만 반영한다.
  - 일반 종료면 각 플레이어의 `won_trick_cards`, 꼴찌 손패 점수, 꼴찌가 이미 딴 트릭 점수를 규칙대로 팀 점수에 반영한다.
  - `apply_round_scores(state, round_state)`가 계산 결과를 `GameState.team_scores`에 누적한다.
- **족보/비교 로직**: `tichu/combo_info.py`
  - `ComboInfo`: 카드 묶음을 해석한 결과(`card_count`, `combo_type`, `strength`, `uses_phoenix`, `resolved_ranks`).
  - `evaluate_combo(cards)`: 카드 목록을 바로 해석해 일반 족보, 봉황 포함 족보, 폭탄을 판정한다.
  - 현재 지원 범위:
    - 일반 족보: 싱글, 페어, 트리플, 풀하우스, 스트레이트, 연속페어
    - 특수 카드: 참새(일반 스트레이트 포함), 용 싱글, 개
    - 봉황 조합: 싱글, 페어, 트리플, 풀하우스, 스트레이트, 연속페어
    - 폭탄: 포카드 폭탄, 스트레이트 플러시 폭탄
  - 내부 구조:
    - 봉황 없는 조합은 `_evaluate_combo_without_phoenix`
    - 봉황 포함 조합은 `_evaluate_combo_with_phoenix`
    - 봉황 스트레이트는 정렬된 숫자를 앞에서부터 따라가며 중간 한 칸 메우기/양끝 확장으로 해석
    - 봉황 연속페어는 `정확히 한 rank만 1장, 나머지는 전부 pair` 조건으로 해석
  - `can_beat(current_combo, selected_combo)`: 기본적으로 같은 종류 족보끼리 비교한다.
    - 봉황 싱글은 직전 싱글 기준 `+0.5`, 단 용 위에는 불가
    - 폭탄은 일반 족보 위에 낼 수 있고, 스트레이트 플러시 폭탄이 포카드 폭탄보다 강하다
- **테스트**: `tichu/test_phase3.py`, `tichu/test_phase4.py`, `tichu/test_phase5_trick_core.py`, `tichu/test_phase6_round_scoring.py`
  - `test_phase3.py`는 일반 족보, 봉황 조합, 개, 폭탄 판정, `can_beat` 비교 규칙까지 검증한다.
  - `test_phase4.py`는 참새 콜 선언/해제, 없음 콜, 빠른 가능성 체크, 봉황 대체, 콜 강제와 패스 가능 여부를 검증한다.
  - `test_phase5_trick_core.py`는 트릭 시작, 카드 제출/패스, 참새 콜 연동, 개, 트릭 종료를 검증한다.
  - `test_phase6_round_scoring.py`는 스몰 티츄, 원투 종료, 일반 종료 점수, 용 전달, 게임 점수 누적을 검증한다.
- **구현 메모**: `tichu/구현_메모.md` — 카드 숫자, 점수, State 분리, 단계별 구현 메모.
