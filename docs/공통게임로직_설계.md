# 공통 게임 로직 설계

> 어떤 보드게임이든 통합적으로 쓸 수 있는 공통 로직. 디버깅·타 보드게임 확장에 유리하게.

**구현 원칙:** 
(1) **추측하여 구현하는 것은 금지**한다. 
(2) 잘 모르거나 애매한 내용이 있으면 **반드시** 사용자에게 질문하고, **명시적인 답을 받기 전에는 해당 부분을 구현하지 않는다.** 
(3) **코드와 함수는 최대한 단순·간결하게** 만든다. 한 함수는 **한 가지 역할만** 담당하고, 불필요한 추상화·분기·중첩을 넣지 않는다. 
(4) **진행 순서**: 보드게임의 실제 진행 순서대로 로직을 구성한다. 
(5) **단계별 구현**: 진행 순서대로 설계하되, 구현은 한 번에 하지 말고 나눠서 진행한다. 
(6) **단계별 계획표**: 각 단계별 계획표에서는 **어떻게 구현할 것인지**를 정확히 명시한다. (예: 사용할 데이터 타입·필드명·함수 시그니처·입출력 형식, 필요한 경우 타입/구조 예시) 그 단계에 속하는 **모든** 함수·로직을 빠짐없이 나열한다. (예: 한 행동이면 입력 수집/선택 생성 → 유효성 검사 → 적용까지, 해당 단계에서 구현할 것은 모두 계획에 포함) 
(7) **계획 단계 역할 명시**: 계획을 세울 때 함수나 코드의 역할을 반드시 주석이나 메모로 명시해 둔다.
(8) **구현 메모**: 해당 보드게임 구현 폴더에는 **md 파일을 따로 두고**, “이 게임을 어떻게 구현했는지”를 메모해 둔다.

---

## 1. 전체 게임 루프

- **한 게임 = 하나의 메인 루프**
- `while` 한 번으로 한 게임 전체를 담당. 기본적으로 **안에 while 중첩 없이** 선턴 → 행동 → 결과 순서로 진행.
- 매 턴/단계마다: 상태 갱신 → 행동 처리 → 종료 여부 확인 → 반복.

---

## 2. 분리할 함수들

아래는 **공통으로 두면 좋은 함수 유형**만 나열. 게임별로 구현/조합.

| 구분 | 설명 | 예시 |
|------|------|-------------|
| **행동(액션)** | 한 번에 하나의 행동만 처리하는 함수 | 카드/말 놓기, 패스, 교환, 선언 등 |
| **점수 계산** | 점수만 계산·반환 (상태 변경 없음) | 턴 점수, 라운드 점수, 보너스 가감 등 |
| **종료 조건** | 게임/라운드/단계 종료 여부 판정 | 목표 점수 도달, 패 소모, 조건 충족 등 |
| **상태 전이** | “지금 단계 → 다음 단계” 결정 | 준비→본게임, 턴/라운드 전환 등 |
| **유효성 검사** | 행동이 규칙에 맞는지만 검사 | 배치 유효, 덮기 가능, 행동 가능 여부 등 |

- 각 **행동/조건/점수**는 **따로 함수로 분리**.
- **상태 처리**: 반환하지 않음. 상태 객체를 인자로 넘기고, 함수 안에서 그 변수(객체)를 수정하는 방식으로 고정.
- **각 함수의 역할은 주석으로 설명·명시한다.** (입력/출력·전제조건·수정하는 상태 등)
- 메인 루프는 이 함수들을 **호출만** 하도록 두면, 디버깅·테스트·다른 게임 확장이 쉬워짐.

---

## 3. 확장·디버깅 포인트

- **통합**: 모든 게임이 같은 “메인 while + 행동/점수/종료 함수” 패턴을 따르면, 로깅·재연·테스트를 한 방식으로 적용 가능.
- **확장**: 새 게임은 “행동 함수 목록”, “점수 함수”, “종료 조건 함수”만 추가/교체.
- **디버깅**: 한 행동·한 조건·한 점수 계산만 단위 테스트하기 좋음.
