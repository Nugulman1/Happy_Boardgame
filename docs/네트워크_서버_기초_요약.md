# 네트워크/서버 기초 요약

## 왜 이 문서를 보는가

- 멀티플레이 게임은 결국 클라이언트와 서버가 상태를 주고받는 구조다.
- 개념 이름이 많아 보여도 초반에는 핵심 관계만 이해하면 된다.
- 이 문서는 보드게임 멀티플레이를 이해하기 위한 최소 개념을 짧게 정리한 복습용 노트다.

## TCP / UDP

### TCP

- 데이터를 순서대로, 빠짐없이 전달하려는 전송 방식이다.
- 재전송, 순서 보장, 연결 유지에 강하다.
- 웹 서비스, 채팅, 턴제 보드게임에 잘 맞는다.

### UDP

- 빠르게 보내는 데 초점을 둔 전송 방식이다.
- 유실 가능성이 있고 순서 보장이 없다.
- 실시간 액션게임, 음성 통신처럼 지연이 더 중요한 경우에 자주 쓴다.

### 보드게임에서 왜 중요한가

- 턴제 보드게임은 속도보다 정합성이 중요하다.
- 카드 한 장, 턴 하나가 빠지면 게임 전체가 깨질 수 있다.
- 그래서 보통 TCP 기반 구조가 잘 맞는다.

## HTTP / WebSocket

### HTTP

- 요청을 보내고 응답을 받는 방식이다.
- 한 번 물어보고 한 번 답받는 구조에 가깝다.
- 생성, 조회, 설정 변경에 적합하다.

### WebSocket

- 연결을 계속 유지하면서 양방향으로 메시지를 주고받는 방식이다.
- 서버도 필요할 때 먼저 메시지를 보낼 수 있다.
- 실시간 게임 진행, 채팅, 알림에 적합하다.

### 보드게임에서 왜 중요한가

- 방 생성, 조회, 시작 같은 작업은 HTTP에 잘 맞는다.
- 플레이, 패스, 턴 전환, 선언 알림은 WebSocket에 잘 맞는다.
- 그래서 둘을 같이 쓰는 경우가 많다.

## 소켓

- 소켓은 네트워크 통신을 위한 연결 창구다.
- 프로그램은 소켓을 통해 데이터를 보내고 받는다.
- `TCP 소켓`, `UDP 소켓`처럼 전송 방식과 함께 생각할 수 있다.

### 보드게임에서 왜 중요한가

- 멀티플레이는 결국 여러 클라이언트와 서버 사이의 연결 관리 문제이기도 하다.
- 연결이 열렸는지, 끊겼는지, 교체됐는지를 다룰 때 소켓 개념이 필요하다.

## 서버 authoritative / state / snapshot / event

### 서버 authoritative

- 서버가 게임 상태의 최종 진실을 가진다는 뜻이다.
- 클라이언트는 입력을 보내고, 서버가 룰을 판정하고 상태를 바꾼다.

### state

- 현재 게임이 어떤 상태인지 나타내는 데이터다.
- 턴, phase, 손패, 점수, 테이블 카드 같은 정보가 포함된다.

### snapshot

- 특정 시점의 state를 클라이언트에 전달하기 위한 복사본이다.
- 클라이언트는 snapshot을 기준으로 화면을 다시 그린다.

### event

- 상태 변화 과정에서 일어난 사건이다.
- 예: 카드가 플레이됨, 턴이 바뀜, 선언이 발생함

### 보드게임에서 왜 중요한가

- 서버 authoritative 구조가 있어야 치트, 오래된 화면, 잘못된 입력을 막기 쉽다.
- snapshot은 정합성을 맞추는 기준이 된다.
- event는 연출, 로그, 알림 표현에 유용하다.

## 세션 / 인증 / 권한

### 세션

- 여러 요청을 같은 사용자 맥락으로 이어서 다루기 위한 개념이다.
- 누가 어느 방에 있었는지, 어떤 좌석인지 이어서 식별할 때 필요하다.

### 인증

- 이 요청을 보낸 사람이 누구인지 확인하는 과정이다.
- 예: 토큰, 로그인, seat token 확인

### 권한

- 그 사람이 이 행동을 해도 되는지 확인하는 과정이다.
- 예: 내 차례인지, host만 시작할 수 있는지

### 보드게임에서 왜 중요한가

- 멀티플레이에서는 누가 어떤 행동을 보내는지 항상 확인해야 한다.
- 인증만 되고 권한이 없을 수도 있다.
- 이 구분이 없으면 남의 턴 조작, 중복 액션, 좌석 탈취 같은 문제가 생긴다.

## 재접속 / stale connection / heartbeat ping pong

### 재접속

- 끊어진 사용자를 원래 게임 맥락으로 다시 붙이는 과정이다.
- 핵심은 연결 복구보다 사용자 식별과 최신 상태 복구다.

### stale connection

- 겉으로는 연결이 남아 있지만 실제로는 더 이상 신뢰하면 안 되는 오래된 연결이다.
- 새 연결이 생겼는데 옛 연결이 남아 있는 상황도 여기에 포함된다.

### heartbeat / ping / pong

- 연결이 아직 살아 있는지 주기적으로 확인하는 방식이다.
- `ping`을 보내고 `pong` 응답이 오면 연결이 살아 있다고 본다.

### 보드게임에서 왜 중요한가

- 멀티플레이에서는 연결이 자주, 그리고 애매하게 끊길 수 있다.
- 재접속이 가능해야 게임을 이어갈 수 있다.
- stale connection 정리와 heartbeat는 연결 상태를 안전하게 관리하는 데 중요하다.

## 게임 서버 아키텍처

- 게임 서버는 보통 통신, 세션 관리, 룰 판정, 상태 직렬화, 저장 계층으로 나눠서 본다.
- 통신 계층은 HTTP/WebSocket 요청을 받는다.
- 세션 계층은 room, player, game session 흐름을 관리한다.
- 룰 계층은 어떤 액션이 합법인지 판단하고 상태를 바꾼다.
- 직렬화 계층은 클라이언트가 보기 좋은 형태로 snapshot을 만든다.

### 보드게임에서 왜 중요한가

- 통신 코드와 룰 코드를 분리해야 테스트와 유지보수가 쉬워진다.
- room과 실제 game session을 구분해야 로비와 게임 진행이 덜 꼬인다.

## 상태 머신

- 상태 머신은 현재 상태에 따라 가능한 행동과 다음 상태가 정해져 있는 구조다.
- 게임 phase, 턴, 라운드 흐름을 안정적으로 관리하는 데 쓰인다.

### 보드게임에서 왜 중요한가

- 아무 때나 아무 액션이 되지 않게 막을 수 있다.
- 준비 단계, 진행 단계, 라운드 종료 단계를 명확히 나눌 수 있다.
- 상태 전이 규칙이 명확할수록 버그를 찾기 쉽다.

## 동시성 / race condition / 중복 요청 / idempotency

### 동시성

- 여러 요청이나 작업이 겹쳐서 처리되는 상황이다.

### race condition

- 처리 순서에 따라 결과가 달라지는 버그다.
- 같은 상태를 여러 요청이 거의 동시에 건드릴 때 자주 생긴다.

### 중복 요청

- 같은 액션이 두 번 이상 들어오는 상황이다.
- 버튼 연타, 재접속, 네트워크 지연, 오래된 연결 때문에 생길 수 있다.

### idempotency

- 같은 요청이 여러 번 들어와도 결과가 한 번 처리한 것처럼 유지되는 성질이다.

### 보드게임에서 왜 중요한가

- 멀티플레이는 동시에 여러 요청이 들어올 수 있다.
- 턴제 게임도 중복 요청과 순서 꼬임은 충분히 발생한다.
- 서버는 현재 authoritative state를 기준으로 다시 검증해야 한다.

## 영속화(DB) / 원자성 / 트랜잭션 / 락

### 영속화(DB)

- 프로그램이 꺼져도 남아 있어야 하는 데이터를 저장하는 것이다.
- 방 정보, 게임 결과, 계정, 전적, 리플레이가 대표 대상이다.

### 원자성

- 어떤 작업이 전부 성공하거나 전부 실패해야 한다는 성질이다.
- 중간 상태만 남으면 안 된다.

### 트랜잭션

- 원자성을 보장하기 위해 작업을 한 묶음으로 처리하는 단위다.

### 락

- 같은 데이터를 동시에 수정하지 못하게 잠깐 막는 장치다.

### 보드게임에서 왜 중요한가

- 메모리 세션만 써도 상태 변경은 한 덩어리처럼 안전하게 처리해야 한다.
- DB가 들어오면 저장 실패와 상태 불일치를 더 조심해야 한다.
- 여러 요청이 같은 room/game를 동시에 수정하지 않게 관리해야 한다.

## 한 줄 정리: 이 프로젝트에 대입하면

- 현재 구조는 `HTTP + WebSocket + FastAPI + Godot + authoritative server + 메모리 세션`으로 볼 수 있다.
- 방/게임 흐름은 서버가 관리하고, 클라이언트는 snapshot과 event를 받아 화면을 갱신한다.
- 지금 단계에서 중요한 것은 새 기술을 늘리는 것보다, 이 구조의 책임 분리를 이해하는 것이다.
