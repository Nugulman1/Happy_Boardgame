# 티츄 구현 메모

이 파일에는 티츄 게임을 **어떻게 구현했는지**(숫자·상수 정의, 타입 선택 등)를 메모해 둡니다.  

---

## 카드 숫자 (내부 표현)

- **일반 트럼프**: **2 ~ 14**만 사용. J/Q/K/A는 별도 구분 없이 11, 12, 13, 14.
- **특수 카드**: 별도 타입 없이 숫자로만 구분.
  - 참새 = **1** (스트레이트에서 1-2-3-4-5처럼 낮은 쪽으로 쓸 수 있음. 역스트레이트 없음)
  - 개 = **20**
  - 봉황 = **21**
  - 용 = **22**

(개·봉황·용은 20~22로 두어 일반 스트레이트 조건 2~14에 안 걸리도록 함.)

---

## 점수 (카드 1장당)

- 5 → 5점  
- 10, K(13) → 10점  
- 봉황(21) → -25점  
- 용(22) → +25점  
- 그 외(참새 1, 개 20 포함) → 0  

---

## State / RoundState (내부·외부 분리)

- **State (GameState)** = **외부만**. 게임 동안 유지·저장. `team_scores`, `round_index` 만. `init_state(state)`는 게임 시작 시 1번만 호출해 이 둘만 초기화.
- **라운드용 (RoundState)** = State가 아님. `hands`, `deck`, `leader_index`, `current_player_index`, `trick_index`, `grand_tichu_declarers`, `current_trick_cards`, `current_trick_combo`, `current_trick_pile`, `last_played_by`, `pass_count_since_last_play`, `mahjong_call_rank`, `small_tichu_declarers`, `played_first_card_players`, `won_trick_cards`, `players_out_order`. **while 라운드 시작할 때마다** `new_round_state()`로 생성해 준비/트릭에 넘김. 내부 항목 추가해도 `init_state` 수정 불필요.
- **3~5단계 트릭/종료/점수**: `current_trick_cards`는 현재 덮어야 하는 최고 조합의 카드만 저장한다. 봉황 싱글처럼 카드만으로는 강도를 복원할 수 없는 경우를 위해 비교용 해석 결과를 `current_trick_combo`에 따로 둔다. 트릭 회수용 전체 카드 목록은 `current_trick_pile`에 따로 모은다. 용으로 트릭을 잡으면 상대 팀 플레이어에게 그 트릭 더미를 넘긴다. 폭탄 비교 규칙은 구현되어 있지만 비턴 폭탄 끼어들기는 아직 막혀 있다.
- **2단계 준비**: 손패는 `sort_hand(hand)`로 배분·교환 **끝**에서 호출. 교환은 4명이 왼쪽/같은팀/오른쪽 줄 3장 선택 후 `apply_exchange(round, choices)`. 라지 티츄는 `round.grand_tichu_declarers`에만 저장, 점수는 5/6단계에서 계산.

## HTTP API v1 메모

### 세션 구조

- HTTP 계층에서는 `GameSession`을 따로 둔다.
- 세션 필드:
  - `game_id`
  - `state: GameState`
  - `round_state: RoundState`
  - `phase`
  - `grand_tichu_responses`
  - `exchange_choices`
  - `last_round_result`
  - `seat_tokens_by_player`
- 즉, **게임 규칙 상태**(`GameState`, `RoundState`)와 **API 오케스트레이션 상태**(`phase`, 집계 중간값, 라운드 결과 보관)를 분리했다.
- `seat_tokens_by_player`는 room에서 시작한 멀티플레이 게임에서만 채우고, 직접 생성한 dev/test 게임에서는 `None`으로 둔다.

### phase 사용 이유

- 현재 API에서는 아래 phase를 쓴다.
  - `prepare_grand_tichu`
  - `prepare_exchange`
  - `trick`
  - `await_dragon_recipient`
  - `game_over`
- 준비 단계와 트릭 단계를 분리하고, 용 수령자 선택이 필요한 경우를 위해 `await_dragon_recipient`를 실제로 둔다.
- 서비스 전이 기준으로는 더 이상 외부용 `round_over` phase를 쓰지 않는다. 라운드 종료 시 결과를 저장한 뒤 바로 다음 라운드 준비 단계 또는 `game_over`로 넘긴다.
- 다만 `session_service.py`의 `Phase` 타입 별칭에는 예전 값 `round_over`가 아직 남아 있으므로, 문서/코드 정리 시 혼동 포인트로 본다.
- Godot는 `phase`를 보고 어떤 UI를 띄울지 결정한다.

### viewer / player_index 분리

- `viewer` = 이번 응답을 누구 시점으로 보여줄지
- `player_index` = 실제 행동한 플레이어
- 서버는 `RoundState.hands` 전체를 들고 있고, 응답에서만 `viewer` 기준으로 숨김 정보를 가공한다.

### 직렬화 규칙

- 카드 JSON은 `{suit, rank}` 형식으로 통일했다.
- 조회 응답은 아래 모양으로 맞췄다.
  - `game_id`
  - `viewer`
  - `phase`
  - `state`
  - `available_actions`
- `state` 안에는:
  - `game`: `team_scores`, `round_index`
  - `table`: `leader_index`, `current_player_index`, `trick_index`, `mahjong_call_rank`, `current_trick_cards`, `current_trick_combo`
  - `players`: 0~3 좌석별 `hand_count`, 티츄 선언 여부, 탈락 여부
  - `viewer_hand`: viewer 본인 카드 상세
  - `players_out_order`
  - `round_result`: 직전 라운드 결과가 있으면 `score_deltas`, `team_scores`, `players_out_order`, `tichu_outcomes`
- 준비 단계에서는 `leader_index`, `current_player_index`를 `null`로 내려 준다.
- 현재 트릭 응답에서는 `current_trick_cards`만 공개하고 `current_trick_pile`은 숨긴다.
- `tichu_outcomes`는 아래 최소 필드로 내려 준다.
  - `player_index`
  - `declaration_type`: `grand | small`
  - `success`

### available_actions

- 현재 최소 필드는 아래를 사용한다.
  - `is_my_turn`
  - `can_play`
  - `can_pass`
  - `can_declare_small_tichu`
  - `can_declare_grand_tichu`
  - `can_choose_dragon_recipient`
- `can_play`, `can_pass`, `can_declare_small_tichu`는 기본적으로 자기 턴일 때만 의미 있게 계산한다.
- `can_choose_dragon_recipient`는 `await_dragon_recipient` 단계에서 용 승자 viewer에게만 켠다.
- exchange 제출 가능 여부는 아직 포함하지 않고 Godot에서 `phase`와 viewer/actor 일치 조건을 함께 본다.
- 폭탄/현재 트릭을 이길 수 있는지 같은 상세 판정은 `play-preview`에서 미리 본다.

### effects

- POST 응답에는 최신 스냅샷과 함께 `effects`를 같이 넣는다.
- `effects`는 엔진 명령이 아니라 “이번 요청으로 발생한 게임 이벤트”만 담는다.
- 현재 사용 중인 effect 예:
  - `game_created`
  - `initial_cards_dealt`
  - `grand_tichu_declared`
  - `remaining_cards_dealt`
  - `cards_exchanged`
  - `phase_changed`
  - `small_tichu_declared`
  - `cards_played`
  - `player_passed`
  - `dragon_recipient_required`
  - `dragon_recipient_chosen`
  - `turn_changed`
  - `trick_won`
  - `round_finished`
  - `game_finished`
- effect별 최소 필드/복구 가능 여부/Godot 의미는 [docs/effects_인벤토리.md](/home/nugulman/dev/Happy_Boardgame/docs/effects_인벤토리.md)에 따로 정리했다.
- `round_finished` effect payload에도 `round_result`와 같은 `tichu_outcomes`를 같이 넣는다. Godot는 이걸 보고 실패 초상화 반응을 낸다.

### Godot presentation 계층

- Godot에서는 서버 `effects`를 UI 코드에 직접 꽂지 않고 아래 순서로 처리한다.
  - `snapshot.effects`
  - `EffectInterpreter.gd`
  - `PresentationBus.gd`
  - `Main.gd` consumer
- `EffectInterpreter.gd`는 현재 서버 effect 15종을 모두 아래 내부 이벤트 이름으로 바꾼다.
  - `announce_game_created`
  - `deal_initial_cards`
  - `announce_grand_tichu`
  - `deal_remaining_cards`
  - `complete_exchange`
  - `update_phase_presentation`
  - `announce_small_tichu`
  - `play_cards_to_table`
  - `announce_pass`
  - `prompt_dragon_recipient`
  - `announce_dragon_recipient`
  - `highlight_current_turn`
  - `collect_trick_cards`
  - `announce_round_finished`
  - `announce_game_finished`
- presentation event 공통 필드는 아래를 사용한다.
  - `name`
  - `source_effect_type`
  - `snapshot_phase`
  - `payload`
  - `requires_visual_change`
  - `snapshot_context`
- `snapshot_context`에는 `game_id`, `viewer`, `phase`, `team_scores`, `round_index`, `current_player_index`, `leader_index`, `trick_index`를 넣어 consumer가 별도 조회 없이 참고할 수 있게 했다.
- `PresentationBus.gd`는 FIFO 큐로만 동작한다. v1은 동시 실행, priority, delay 없이 한 이벤트를 처리 완료한 뒤 다음 이벤트를 보낸다.
- 미구현 이벤트도 bus에 들어가지만, 현재 consumer가 없으면 no-op로 ack만 하고 지나간다. 즉 구조는 고정하고 연출만 나중에 추가하는 방식이다.

  - 현재 턴 플레이어는 해당 패널 하이라이트 배경으로 강조한다.
  - `turn_changed` effect가 없는 최신 snapshot을 받을 때도 `current_player_index`로 현재 강조 상태를 복구한다.
- 선언 연출
  - `DeclarationBanner` 패널에 `Player N / Grand Tichu`, `Player N / Small Tichu` 같은 문구를 잠깐 띄운다.
  - 선언한 플레이어 초상화 placeholder는 `declared` 상태로 잠깐 바뀐다.
  - grand tichu 응답에서 `declare=false`이면 큰 배너나 선언 초상화 전환을 하지 않는다.
- 카드 이동/트릭 회수 연출
  - 플레이어에서 중앙으로 날아가는 모션은 쓰지 않는다.
  - 대신 중앙 `TableContainer`가 snapshot 갱신 시 짧게 강조되어 카드가 자연스럽게 놓인 것처럼 보이게 한다.
  - 중앙 테이블 카드는 단순 문자열 목록 대신 수트/랭크별 색을 입힌 카드 칩으로 그린다.
  - 손패도 같은 기준의 버튼 스타일을 쓰고, 선택된 카드는 테두리/배경 대비를 더 높여 표시한다.
  - `trick_won`도 카드 회수 문구 대신 중앙 카드 영역 정리 + 짧은 강조로 끝낸다.
- 라운드 점수 연출
  - `ResultOverlay` 패널은 `round_finished`, `game_finished`를 구분해서 제목/본문을 보여 준다.
  - 오버레이는 현재 placeholder 톤을 고정해 라운드 종료와 게임 종료가 색/문구에서 구분되도록 둔다.
  - `ScoreSummaryLabel`에는 `Team 0/2`, `Team 1/3` 기준 현재 총점과 이번 라운드 delta를 계속 남긴다.
  - `round_finished`의 `tichu_outcomes.success == false`인 플레이어는 초상화가 `failed` 상태로 잠깐 바뀐다.
- 초상화 placeholder
  - 각 플레이어 패널에는 `PortraitSlot`이 있고, 현재는 실제 이미지 대신 색상/문구로 상태를 보여 준다.
  - 상태는 `base`, `declared`, `failed` 3가지다.
  - 라벨 문구는 `P0 / BASE SLOT` 형태로 두고, tooltip에는 `player_0_base.png` 같은 자산 슬롯 이름을 보여 준다.
  - 나중에 실제 2D 이미지가 준비되면 같은 위치에서 texture 교체로 확장할 수 있게 만들었다.
- 위 연출은 모두 placeholder다. 나중에 같은 consumer 위치를 실제 이미지, Tween, SFX, 캐릭터 리액션으로 교체하는 구조를 의도했다.

### PresentationBus ACK 메모

- 초기 구현은 presentation event 시작 직후 바로 `acknowledge_current()`를 호출하는 방식이었다.
- 선언 배너, 결과 패널, 초상화 상태처럼 `잠깐 보여주고 사라져야 하는 연출`은 즉시 ACK로는 표현이 안 된다.
- 현재는 `Main.gd`가 필요한 이벤트만 비동기 ACK를 잡고, 타이머가 끝난 뒤 `acknowledge_current()`를 호출한다.
- 그래서 선언 배너/결과 패널이 표시 시간을 가진 뒤 다음 presentation event로 넘어간다.

### WebSocket 액션 + snapshot 구조

- 현재 인게임 계층은 아래 구조다.
  - 액션 입력: `WS /ws/games/{game_id}?viewer=...&seat_token=...`
  - 상태 동기화: 같은 game socket의 `snapshot` 이벤트
  - helper API: `GET /legal-plays`, `POST /play-preview`는 HTTP 유지
- `SocketManager`가 `game_id -> viewer -> websocket` 연결을 메모리로 관리한다.
- 같은 viewer가 다시 접속하면 이전 연결을 닫고 새 연결로 교체한다.
- WebSocket 연결 직후 서버는 `type="snapshot"` 이벤트로 최신 snapshot을 1회 내려준다.
- 클라이언트 액션 메시지 형식은 아래다.
  - `{"type":"action","request_id":1,"action":"play","payload":{...}}`
- 서버 응답은 아래처럼 나눈다.
  - 성공: `action_result`
  - 실패: `action_error`
  - 상태 반영: `snapshot`
- 성공 시 순서는 `action_result`를 요청자에게 먼저 보내고, 그 다음 연결된 viewer 전체에 `snapshot`을 브로드캐스트한다.
- 손패와 `available_actions`가 viewer마다 다르므로 snapshot은 viewer별로 다시 직렬화한다.
- 상태의 단일 소스는 여전히 `snapshot`이다. `action_result`는 성공 확인용이고 상태 전체를 담지 않는다.

### room / seat token 멀티플레이 계층

- 게임 세션 위에 room 계층을 별도로 얹었다.
- `RoomSession` 필드:
  - `room_code`
  - `status`: `lobby | in_game | finished`
  - `host_seat_index`
  - `seats[4]`
  - `game_id`
- `SeatState` 필드:
  - `seat_index`
  - `claimed`
  - `connected`
  - `seat_token`
  - `last_seen_at`
- room 생성 시 `seat 0`이 자동으로 host가 된다.
- join은 빈 좌석만 허용한다. 점유된 좌석은 `409`로 막는다.
- host가 `start`를 호출하면 현재 seat token 맵을 `GameSession.seat_tokens_by_player`에 복사한 뒤 실제 게임 세션을 만든다.
- room socket은 `WS /ws/rooms/{room_code}?seat_token=...`를 사용하고, 로비 snapshot만 브로드캐스트한다.
- 게임 시작 후에는 room snapshot으로 좌석 상태를 보여주고, 실제 인게임 동기화는 기존 game socket을 재사용한다.

### room-backed 게임 인증 규칙

- direct dev/test 게임(`POST /games/tichu`)은 기존처럼 seat token 없이 동작한다.
- room-backed 게임은 `seat_token -> player_index` 인증을 강제한다.
- 적용 위치:
  - `GET /games/{game_id}`
  - `WS /ws/games/{game_id}`
  - 준비/액션 POST
  - `GET /legal-plays`
  - `POST /play-preview`
- viewer 조회는 `seat_token`과 같은 좌석 번호만 허용한다.
- action payload의 `player_index`도 인증된 좌석 번호와 같아야 한다.
- 즉 실제 멀티플레이 경로에서는 `viewer`나 `player_index`를 임의로 바꿔 남의 손패를 보거나 대신 행동할 수 없게 했다.

### 준비 단계 오케스트레이션

- API 계층에서는 `run_prepare_phase()`를 직접 호출하지 않는다.
- 대신 준비 단계를 엔드포인트에서 나눠 처리한다.
  1. `POST /games/tichu` → 새 세션, 8장 배분, `prepare_grand_tichu`
  2. `POST /prepare/grand-tichu` → 4명 응답 집계, 완료 시 6장 추가 배분
  3. `POST /prepare/exchange` → 4명 교환 선택 집계, 완료 시 `apply_exchange`, `set_leader_by_mahjong`, `start_trick`

### 액션 처리

- `POST /actions/small-tichu`는 `declare_small_tichu()`를 그대로 감싼다.
- `POST /actions/play`는 `play_cards()`를, `POST /actions/pass`는 `pass_turn()`를 감싼다.
- 용이 현재 최고 카드인 상태에서 마지막 패스로 트릭이 끝나면 즉시 정산하지 않고 `await_dragon_recipient`로 멈춘다.
- `POST /actions/dragon-recipient`에서 용 승자가 상대 팀 recipient를 고르면 그때 `resolve_trick_end()`를 실행한다.
- 액션 후 라운드가 종료되면 `finalize_round()`를 호출하고, 게임이 끝나지 않았으면 즉시 다음 라운드 준비 단계(`prepare_grand_tichu`)로 넘긴다.
- 직전 라운드 결과는 `last_round_result`에 남겨 다음 snapshot에서도 같이 보여줄 수 있게 했다.

### 보조 API

- `GET /legal-plays`는 현재 트릭 기준 합법 조합을 그대로 반환한다. 주 용도는 참새 콜 대응 확인이다.
- `POST /play-preview`는 현재 세션 상태 기준으로 선택 카드가 실제 제출 가능한지 미리 계산한다.
  - `beats_current_trick`
  - `satisfies_mahjong_call`
  - `can_submit_play`
  - `reason_code`, `message`
  - `selected_combo`, `current_trick_combo`
- 폭탄 끼어들기 액션은 아직 구현하지 않았고, 현재는 조합 판정/UI 보조만 지원한다.

## 3단계 족보/비교 구현 메모

### `ComboInfo` 구조

- `card_count`: 카드 장수
- `combo_type`: 판정된 족보 이름
- `strength`: 같은 종류 족보끼리 비교할 값
- `uses_phoenix`: 봉황 사용 여부
- `resolved_ranks`: 봉황이 실제 어떤 숫자 조합으로 해석되었는지 기록

현재 `combo_type`은 아래를 사용한다.

- 일반 족보: `single`, `pair`, `triple`, `full_house`, `straight`, `pair_run`
- 폭탄: `bomb_four`, `bomb_straight_flush`

### `evaluate_combo(cards)` 구조

- 입력 카드 정렬 후 봉황 포함 여부를 먼저 분기한다.
- 봉황이 없으면 `_evaluate_combo_without_phoenix`
- 봉황이 있으면 `_evaluate_combo_with_phoenix`

이렇게 분리해 두어서 일반 족보 판정과 봉황 해석 로직이 섞이지 않게 했다.

### 일반 족보/특수 카드

- 참새는 rank `1`로 취급한다.
- 용은 `single`만 허용한다.
- 개는 `dog`라는 별도 특수 조합으로 판정한다.
- 일반 조합에서는 폭탄을 먼저 판정한 뒤, 아니면 일반 족보로 내려간다.

### 봉황 구현 범위

- 지원:
  - `single`
  - `pair`
  - `triple`
  - `full_house`
  - `straight`
  - `pair_run`
- 봉황과 함께 낼 수 없는 특수 카드는 `개`, `참새`, `용`으로 고정했다.

### 봉황 싱글 비교 규칙

- 봉황 싱글 자체 판정은 `strength=(0.5,)`로 저장한다.
- 실제 비교는 `can_beat()`에서 처리한다.
- 현재 싱글 위에 낼 때는 `직전 rank + 0.5`로 계산한다.
- 용 위에는 봉황 싱글을 낼 수 없게 막아 두었다.
- 한 번 테이블 위에 올라간 봉황 싱글은 그 강도가 유지돼야 하므로, `play_cards()`가 `current_trick_combo`에 실제 비교 강도(`10.5` 같은 값)를 저장한다.
- 참새 콜/합법 조합 탐색도 이 내부 `current_trick_combo`를 우선 사용해서 봉황 싱글 위 판정이 틀어지지 않게 했다.

### 봉황 스트레이트 해석

- 봉황을 제외한 일반 카드 숫자를 정렬해서 앞에서부터 따라간다.
- 차이가 `1`이면 그대로 진행한다.
- 차이가 `2`이면 봉황으로 중간 숫자 한 칸을 메운다.
- 차이가 `3` 이상이면 실패한다.
- 중간에서 봉황을 안 썼으면 양끝 확장을 시도한다.
  - 위쪽 확장을 우선한다.
  - 위쪽이 막히면 아래쪽 확장을 본다.
- 봉황이 `1`을 대체하는 경우는 허용하지 않는다.
- 최종 확정된 숫자열은 `resolved_ranks`에 저장한다.

예:

- `2,3,5,6 + 봉황 -> 2,3,4,5,6`
- `4,5,6,7,8 + 봉황 -> 4,5,6,7,8,9`
- `10,J,Q,K,A + 봉황 -> 9,10,11,12,13,14`

### 봉황 연속페어 해석

- 카드 수는 4장 이상, 짝수여야 한다.
- 봉황 제외 일반 카드의 rank 개수를 센다.
- 허용 패턴은 `정확히 한 rank만 1장`, `나머지는 모두 2장`이다.
- unique rank가 연속이어야 한다.
- 봉황은 부족한 1장을 채워 pair를 완성하는 용도로만 쓴다.
- 비어 있는 새 pair 전체 생성이나 양끝 확장은 허용하지 않는다.

예:

- `33 4 + 봉황 -> 33 44`
- `33 44 5 + 봉황 -> 33 44 55`
- `33 55 + 봉황`은 실패

### 폭탄 구현

- 포카드 폭탄:
  - `combo_type="bomb_four"`
  - `strength=(rank,)`
- 스트레이트 플러시 폭탄:
  - `combo_type="bomb_straight_flush"`
  - `strength=(length, high_rank)`

폭탄 판정 규칙:

- 포카드: 4장 같은 rank
- 스트레이트 플러시: 5장 이상, 같은 suit, 연속 숫자
- 봉황 포함 폭탄은 불가
- 특수카드는 현재 카드 설계상 suit/rank 조건에서 자동으로 배제된다

### `can_beat()` 비교 규칙

- 기본은 같은 종류 족보끼리 `strength` 비교
- 폭탄은 일반 족보 위에 낼 수 있다
- 일반 족보는 폭탄을 못 이긴다
- 포카드 폭탄끼리는 rank 비교
- 스트레이트 플러시 폭탄끼리는 길이 우선, 길이가 같으면 최고 rank 비교
- 스트레이트 플러시 폭탄이 모든 포카드 폭탄보다 강하다

### 테스트

- `tichu/test_phase3.py`에서 아래를 검증한다.
  - 일반 족보 판정
  - 개(`dog`) 판정
  - 봉황 싱글/페어/트리플/풀하우스/스트레이트/연속페어
  - 폭탄 판정
  - `can_beat()`의 봉황 비교 규칙
  - `can_beat()`의 폭탄 예외 비교 규칙

## 3단계 참새 소원/콜 구현 메모

### 상태와 저장 방식

- 현재 라운드의 참새 콜은 `RoundState.mahjong_call_rank` 하나로 저장한다.
- `None`은 두 의미를 함께 가진다.
  - 아직 콜이 없음
  - 참새를 냈지만 숫자를 부르지 않고 `없음`을 선택함
- 지금 단계에서는 트릭 루프가 아직 없어서, 위 두 상태를 구분하는 별도 필드는 두지 않았다.

### 선언 규칙

- `can_declare_mahjong_call(played_cards, call_rank)`는 아래 3개를 확인한다.
  - 참새가 실제로 포함되어 있는가
  - `played_cards`가 `evaluate_combo()` 기준 합법 조합인가
  - `call_rank`가 `없음(None)` 또는 `2~14`인가
- `set_mahjong_call(round_state, played_cards, call_rank)`는 위 검사를 통과한 경우만 저장한다.
- 잘못된 선언은 `ValueError`로 막는다.

### 콜 충족 가능성 빠른 검사

- `hand_can_possibly_match_mahjong_call(hand, call_rank)`는 정밀 탐색 전에 cheap pre-check로 사용한다.
- 손패에 아래 둘 중 하나라도 있으면 `True`다.
  - 실제 콜 숫자 카드
  - 봉황
- 둘 다 없으면 그 턴에는 콜 충족 조합을 만들 수 없으므로 subset 전체 탐색을 건너뛴다.

### 합법 조합 탐색 방식

- `find_legal_plays(hand, current_trick_cards)`는 손패의 non-empty subset을 전부 생성해서 검사한다. 이 탐색은 주로 참새 콜 강제 여부 판단에 사용한다.
- 각 subset마다:
  - `evaluate_combo(selected_cards)`로 유효 족보인지 확인
  - 현재 트릭이 있으면 `can_beat(current_combo, selected_combo)`로 이길 수 있는지 확인
- 즉, “족보 생성기”를 따로 두지 않고 현재는 브루트포스 방식으로 모든 합법 조합을 찾는다.
- 14장 기준 최악의 경우 subset 수는 `2^14 - 1 = 16383`개라서, 프론트/엔진 연동 시에는 매 프레임 호출 대신 이벤트 기반 호출이나 캐시가 필요하다.

### 콜 충족 판정

- `selection_satisfies_mahjong_call(selected_cards, call_rank)`는 아래 순서로 판정한다.
  - 콜이 없으면 항상 `True`
  - 선택 카드가 합법 조합이 아니면 `False`
  - 실제 콜 숫자 카드가 포함되어 있으면 `True`
  - 봉황 조합이면 `ComboInfo.resolved_ranks` 안에 콜 숫자가 있으면 `True`
- 즉, 봉황은 콜 숫자를 직접 대신할 수 있고, 그 판단은 별도 감으로 하지 않고 `evaluate_combo()`의 최종 해석 결과와 일치시킨다.

예:

- 콜 7, `7 + 봉황` 페어 → 충족
- 콜 7, `8 + 봉황` 페어 → 불충족
- 콜 9, `8, 봉황, 10, J, Q` 스트레이트 → 충족

### 강제 규칙

- `find_legal_plays_matching_call()`는 합법 조합 중에서 콜까지 만족하는 조합만 다시 걸러낸다.
- `must_follow_mahjong_call()`는 `find_legal_plays_matching_call()` 결과가 하나라도 있으면 `True`다.
- `can_pass_with_mahjong_call()`는 위 값의 반대로 구현했다.
- 트릭 액션 쪽에서는 일반 플레이 상황에 `find_legal_plays()` 전체 탐색을 쓰지 않고, 선택한 카드 조합만 직접 판정한다.
- 즉, 현재 손패와 트릭 상태에서 콜을 만족하는 합법 조합이 실제로 존재할 때만 패스를 막는다.

## 4~5단계 종료/점수 구현 메모

- `declare_small_tichu(round_state, player_index)`는 플레이어가 첫 카드 제출 전에만 선언을 저장한다.
- `played_first_card_players`는 `play_cards()` 성공 시 갱신한다.
- 개는 `play_cards()` 안에서 일반 카드 제출과 분기한다.
  - 현재 트릭이 비어 있어야 한다.
  - 현재 트릭의 선 플레이어가 개를 냈을 때만 허용한다.
  - 제출 성공 시 트릭 더미를 만들지 않고 같은 팀 플레이어에게 즉시 선을 넘긴다.
- 용은 즉시 recipient를 받지 않는다.
  - 마지막 패스로 용 트릭이 확정되면 `await_dragon_recipient` 단계로 넘어간다.
  - 이 단계에서는 용 승자만 recipient를 고를 수 있다.
  - recipient 확정 후에만 `resolve_trick_end()`로 실제 더미 전달과 다음 선 계산을 진행한다.
- `is_double_victory()`는 `players_out_order`의 앞 2명이 같은 팀인지로 판정한다.
- 일반 종료는 `3등이 마지막 패를 낸 뒤 현재 트릭이 끝난 시점`에만 성립한다. 그래서 `resolve_trick_end()`는 3등 이후에는 새 트릭을 열지 않고 현재 트릭 상태만 비운다.
- `calculate_round_score_deltas()`는 아래 순서로 계산한다.
  - 원투면 기본 200점과 티츄 선언 점수만 반영
  - 일반 종료면 플레이어별 획득 트릭 점수를 합산
  - 꼴찌 손패 점수는 1등 팀으로 이동
  - 꼴찌가 이미 따둔 트릭 점수는 상대 팀으로 이동
- `run_game(state)`는 아직 자동 플레이 루프가 아니다.
  - 게임 상태를 초기화하고 새 라운드 준비 단계까지만 진행한다.
  - 실제 트릭 액션 반복은 외부가 `play_cards()` / `pass_turn()`로 넣는다.
  - 라운드 종료 뒤 `finalize_round(state, round_state)`를 호출해 점수를 누적하는 구조다.

## Godot 1차 클라이언트 메모

- 구조:
  - `ApiClient.gd`: HTTP 요청 + WebSocket 연결/폴링 래퍼
  - `GameStore.gd`: 최신 snapshot, `legal_plays`, `play_preview`, 선택 viewer/actor 저장
  - `Main.gd`: 메인 UI와 입력 흐름
- 현재 Godot는 테스트용 단일 화면에서 아래를 확인할 수 있다.
  - `/health`
  - 새 게임 생성 / 상태 새로고침
  - viewer 기준 WebSocket 연결 / 실시간 snapshot 반영
  - viewer / actor 분리 테스트
  - grand tichu
  - exchange
  - small tichu
  - play / pass
  - dragon recipient 선택
  - 참새 `call_rank` 선택
  - 현재 `mahjong_call_rank` 표시
  - `state.players`, `leader_index`, `trick_index`, `players_out_order` 표시
  - `effects` 로그 표시
  - `legal-plays` 조회
  - `play-preview` 기반 제출 가능 여부/사유 표시
  - HTTP 액션 응답과 WebSocket push 중복 반영 방지
  - 자동 다음 라운드 / game over 표시
- 현재 주요 액션 버튼은 `available_actions`를 기준으로 켜고 끈다.
  - `play`
  - `pass`
  - `grand tichu`
  - `small tichu`
  - `dragon recipient`
- 다만 exchange 제출 가능 여부는 아직 서버 필드 없이 `phase == prepare_exchange`와 viewer/actor 일치 조건을 로컬에서 함께 본다.
- 아직 Godot에 직접 안 붙은 백엔드 기능:
  - 액션 WebSocket 제출
- 실제 멀티플레이 클라이언트가 붙으면 `viewer`는 각 클라이언트마다 고정되고, 지금의 `actor` 분리 UI는 테스트 전용으로만 남는다.
