# 티츄 구현 메모

이 파일에는 티츄 게임을 **어떻게 구현했는지**(숫자·상수 정의, 타입 선택 등)를 메모해 둡니다.  

---

## 카드 숫자 (내부 표현)

- **일반 트럼프**: **2 ~ 14**만 사용. J/Q/K/A는 별도 구분 없이 11, 12, 13, 14.
- **특수 카드**: 별도 타입 없이 숫자로만 구분.
  - 참새 = **1** (스트레이트에서 1-2-3-4-5처럼 낮은 쪽으로 쓸 수 있음. 역스트레이트 없음)
  - 개 = **20**
  - 봉황 = **21**
  - 용 = **22**

(개·봉황·용은 20~22로 두어 일반 스트레이트 조건 2~14에 안 걸리도록 함.)

---

## 점수 (카드 1장당)

- 5 → 5점  
- 10, K(13) → 10점  
- 봉황(21) → -25점  
- 용(22) → +25점  
- 그 외(참새 1, 개 20 포함) → 0  

---

## State / RoundState (내부·외부 분리)

- **State (GameState)** = **외부만**. 게임 동안 유지·저장. `team_scores`, `round_index` 만. `init_state(state)`는 게임 시작 시 1번만 호출해 이 둘만 초기화.
- **라운드용 (RoundState)** = State가 아님. `hands`, `deck`, `leader_index`, `trick_index`, `grand_tichu_declarers`. **while 라운드 시작할 때마다** `new_round_state()`로 생성해 준비/트릭에 넘김. 내부 항목 추가해도 `init_state` 수정 불필요.
- **2단계 준비**: 손패는 `sort_hand(hand)`로 배분·교환 **끝**에서 호출. 교환은 4명이 왼쪽/같은팀/오른쪽 줄 3장 선택 후 `apply_exchange(round, choices)`. 라지 티츄는 `round.grand_tichu_declarers`에만 저장, 점수는 5/6단계에서 계산.

## 3단계 족보/비교 구현 메모

### `ComboInfo` 구조

- `card_count`: 카드 장수
- `combo_type`: 판정된 족보 이름
- `strength`: 같은 종류 족보끼리 비교할 값
- `uses_phoenix`: 봉황 사용 여부
- `resolved_ranks`: 봉황이 실제 어떤 숫자 조합으로 해석되었는지 기록

현재 `combo_type`은 아래를 사용한다.

- 일반 족보: `single`, `pair`, `triple`, `full_house`, `straight`, `pair_run`
- 폭탄: `bomb_four`, `bomb_straight_flush`

### `evaluate_combo(cards)` 구조

- 입력 카드 정렬 후 봉황 포함 여부를 먼저 분기한다.
- 봉황이 없으면 `_evaluate_combo_without_phoenix`
- 봉황이 있으면 `_evaluate_combo_with_phoenix`

이렇게 분리해 두어서 일반 족보 판정과 봉황 해석 로직이 섞이지 않게 했다.

### 일반 족보/특수 카드

- 참새는 rank `1`로 취급한다.
- 용은 `single`만 허용한다.
- 개는 현재 족보 판정에서 제외한다.
- 일반 조합에서는 폭탄을 먼저 판정한 뒤, 아니면 일반 족보로 내려간다.

### 봉황 구현 범위

- 지원:
  - `single`
  - `pair`
  - `triple`
  - `full_house`
  - `straight`
  - `pair_run`
- 봉황과 함께 낼 수 없는 특수 카드는 `개`, `참새`, `용`으로 고정했다.

### 봉황 싱글 비교 규칙

- 봉황 싱글 자체 판정은 `strength=(0.5,)`로 저장한다.
- 실제 비교는 `can_beat()`에서 처리한다.
- 현재 싱글 위에 낼 때는 `직전 rank + 0.5`로 계산한다.
- 용 위에는 봉황 싱글을 낼 수 없게 막아 두었다.

### 봉황 스트레이트 해석

- 봉황을 제외한 일반 카드 숫자를 정렬해서 앞에서부터 따라간다.
- 차이가 `1`이면 그대로 진행한다.
- 차이가 `2`이면 봉황으로 중간 숫자 한 칸을 메운다.
- 차이가 `3` 이상이면 실패한다.
- 중간에서 봉황을 안 썼으면 양끝 확장을 시도한다.
  - 위쪽 확장을 우선한다.
  - 위쪽이 막히면 아래쪽 확장을 본다.
- 봉황이 `1`을 대체하는 경우는 허용하지 않는다.
- 최종 확정된 숫자열은 `resolved_ranks`에 저장한다.

예:

- `2,3,5,6 + 봉황 -> 2,3,4,5,6`
- `4,5,6,7,8 + 봉황 -> 4,5,6,7,8,9`
- `10,J,Q,K,A + 봉황 -> 9,10,11,12,13,14`

### 봉황 연속페어 해석

- 카드 수는 4장 이상, 짝수여야 한다.
- 봉황 제외 일반 카드의 rank 개수를 센다.
- 허용 패턴은 `정확히 한 rank만 1장`, `나머지는 모두 2장`이다.
- unique rank가 연속이어야 한다.
- 봉황은 부족한 1장을 채워 pair를 완성하는 용도로만 쓴다.
- 비어 있는 새 pair 전체 생성이나 양끝 확장은 허용하지 않는다.

예:

- `33 4 + 봉황 -> 33 44`
- `33 44 5 + 봉황 -> 33 44 55`
- `33 55 + 봉황`은 실패

### 폭탄 구현

- 포카드 폭탄:
  - `combo_type="bomb_four"`
  - `strength=(rank,)`
- 스트레이트 플러시 폭탄:
  - `combo_type="bomb_straight_flush"`
  - `strength=(length, high_rank)`

폭탄 판정 규칙:

- 포카드: 4장 같은 rank
- 스트레이트 플러시: 5장 이상, 같은 suit, 연속 숫자
- 봉황 포함 폭탄은 불가
- 특수카드는 현재 카드 설계상 suit/rank 조건에서 자동으로 배제된다

### `can_beat()` 비교 규칙

- 기본은 같은 종류 족보끼리 `strength` 비교
- 폭탄은 일반 족보 위에 낼 수 있다
- 일반 족보는 폭탄을 못 이긴다
- 포카드 폭탄끼리는 rank 비교
- 스트레이트 플러시 폭탄끼리는 길이 우선, 길이가 같으면 최고 rank 비교
- 스트레이트 플러시 폭탄이 모든 포카드 폭탄보다 강하다
